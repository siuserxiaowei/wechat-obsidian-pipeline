# Hermes 本机只读微信 + Obsidian 日程卡片工作流

这份文档是给“只有一台主力机、一个主微信”的人看的。

之前这套仓库更偏自动化：`wx-cli / WeFlow` 拉数据，写进 Obsidian，再生成群日报。现在加一条更轻的路线：**Hermes 只读当前微信窗口，帮你看消息、提炼待办、整理链接和文档，最后写成 Obsidian 日程卡片。**

它不是微信机器人。

更准确地说，它是一个坐在你旁边的桌面助理。你把微信窗口打开，它帮你读；它可以写草稿，但不替你点发送。

## 先说清楚：Hermes 跑在哪里

当前安装方式是本机运行。

```text
Mac Terminal
  -> Hermes Agent
  -> OpenAI Codex / ChatGPT 模型
  -> 读取你当前给它看的内容
```

它不在 VPS 上。

你关掉 Terminal，Hermes 就停。你重启电脑，它也不会自己回来，除非你额外配置开机启动。

这点和很多 OpenClaw / 服务器 bot 不一样。OpenClaw 常见做法是放在国外 VPS 上，优点是 24 小时在线；但 VPS 读不到你 Mac 上的桌面微信窗口，也读不到本机微信数据库。想处理本机微信，终究要有一个本机执行端。

## 要不要每天开关

分三种情况。

| 用法 | 要不要每天开 |
|---|---|
| 只在电脑前临时看微信 | 要，用的时候打开 Terminal 跑 Hermes |
| 希望开机后自动准备好 | 可以做 macOS 登录启动 |
| 希望像机器人一样 24 小时在线 | 需要 gateway / VPS / 桥接服务，但主微信风险会上升 |

我现在建议第一种或第二种。

原因很简单：主力机 + 主微信，先别让它后台常驻接管。先把“读消息、写卡片”跑顺，比一上来搞全自动更稳。

本机启动命令可以是：

```bash
cd "/Users/siuserxiaowei/Documents/New project 3/00_Hermes微信只读秘书_2026-05-10/hermes-workspace"
~/.local/bin/hermes chat
```

也可以包成脚本：

```bash
#!/usr/bin/env bash
set -euo pipefail

cd "/Users/siuserxiaowei/Documents/New project 3/00_Hermes微信只读秘书_2026-05-10/hermes-workspace"
exec "$HOME/.local/bin/hermes" chat
```

## 只读边界

这条线很重要。

允许：

- 读当前微信窗口里可见的信息。
- 总结重点消息。
- 提取待办、链接、文件名、会议时间。
- 草拟回复。
- 整理成 Obsidian 卡片。

不允许：

- 自动发送微信。
- 自动转发、收藏、删除、撤回。
- 自动加好友、拉群、退群。
- 扫码绑定未知服务。
- 保存微信 token 或二维码。
- 把完整聊天记录上传到公开仓库。

最终发送，还是人手动发。

## 链接和文档为什么有时读不了

Hermes 只看微信窗口时，看到的是“屏幕上显示的文字”。它能读到：

- 群名
- 发言人
- 当前可见消息
- 链接标题或 URL
- 文件名

但它不一定能直接读到：

- 链接打开后的正文
- 飞书文档里的内容
- 微信文件里的 Word / PDF / Excel 正文
- 图片里的小字

不是它不聪明，是权限和数据通道没给到。

## 读取链接的三种方法

### 方式一：复制链接给 Hermes

最稳。

你在微信里复制链接，然后发给 Hermes：

```text
读一下这个链接，帮我整理成今天的 Obsidian 日程卡片：
https://example.com/...
```

如果 Hermes 当前有 web / browser 工具，它可以尝试打开和提取正文。

如果链接需要登录，比如飞书、腾讯文档、企业微信文档，那就更适合方式二。

### 方式二：用浏览器打开，给 Hermes 看页面

适合登录态页面。

你先用自己浏览器打开链接，确认能看到正文，再让 Hermes / Codex 读取当前页面或截图。

优点是不用把账号密码交给机器人。缺点是自动化程度低一点。

### 方式三：下载文档到本地，再让 Hermes 读文件

适合 PDF、Word、Excel、PPT。

流程：

```text
微信收到文件
  -> 下载到本机
  -> 放到 Hermes 工作目录或 Obsidian 附件目录
  -> Hermes 读取文件
  -> 摘要写入日程卡片
```

这样最好控风险。文档留在本机，整理结果再进 Obsidian。

## 文档进入 Obsidian 的建议结构

推荐继续用这个目录：

```text
Obsidian Vault/
└── 微信渠道/
    ├── 日程卡片/
    │   └── 2026-05/
    │       └── 2026-05-10.md
    ├── 文件传输助手/
    ├── 公众号与链接/
    └── attachments/
```

日程卡片不是原始聊天记录。它只放整理后的结果：

- 今天谁找我
- 哪些消息需要回
- 哪些文件要看
- 哪些链接值得收藏
- 哪些事情可以延后

原始聊天能不放就不放。尤其是主微信里的私人聊天，不要为了自动化把隐私堆进库里。

## 一张日程卡片应该长什么样

见模板：

```text
templates/obsidian_daily_card.md.j2
```

核心字段：

- `source`: 来源，比如微信、飞书、群聊、文件传输助手。
- `privacy`: 隐私等级。
- `status`: 今天是否处理完。
- `people`: 相关人员。
- `links`: 今日链接。
- `files`: 今日文件。
- `reply_drafts`: 待回复草稿。

卡片内容可以这样：

```markdown
# 2026-05-10 微信日程卡片

## 今天要处理

- [ ] 回复 A：确认明晚会议时间
- [ ] 看 B 发来的飞书文档
- [ ] 把 C 的需求整理成项目 todo

## 重要消息

### A

他说下周可以对接，但要先看方案。

建议回复：

> 我今晚整理一下，明天上午发你一版。

## 链接与文档

- 飞书会议纪要：待读
- 项目报价表：已下载，待整理
```

## 和原来的群日报怎么结合

原来的 `wx-cli / WeFlow` 适合批量、定时、无人值守。

Hermes 只读模式适合人机协作：

```text
微信群日报：每天自动扫一遍群
微信日程卡片：你坐在电脑前，让 Hermes 帮你处理眼前消息
```

两者可以共存。

推荐分工：

| 类型 | 用什么 |
|---|---|
| 付费群、公开群、信息流很多 | `wx-cli / WeFlow` 日报 |
| 主微信私聊、临时文件、当天待办 | Hermes 只读卡片 |
| 链接、文档、会议纪要 | 浏览器 / 本地文件解析后进 Obsidian |

## 一句话工作流

```text
打开微信
  -> Hermes 只读当前窗口
  -> 复制需要读的链接 / 下载需要读的文件
  -> Hermes 提炼待办和草稿
  -> 写入 Obsidian 日程卡片
  -> 人手动回复微信
```

## 最小可用提示词

把微信停在你要处理的窗口，然后对 Hermes 说：

```text
你现在是我的微信只读秘书。

请只根据当前微信窗口和我提供的链接/文件来整理，不要自动发送任何微信消息。

请输出：
1. 今天必须处理的消息
2. 可以延后的消息
3. 需要我手动回复的人
4. 每个人的回复草稿
5. 需要进入 Obsidian 的链接和文档
6. 最后生成一份 Obsidian 日程卡片 Markdown
```

如果要读链接：

```text
这是微信里复制出来的链接，请先读正文，再合并进今天的日程卡片：
<URL>
```

如果要读文件：

```text
这个文件是微信里下载下来的，请提取重点、待办和可回复内容，合并进今天的日程卡片：
<本地文件路径>
```

## 公开分享时怎么讲

不要讲成“AI 接管主微信”。

更安全、更准确的说法是：

> 我没有让 AI 自动回微信，而是做了一个本机只读秘书。它只帮我看当前微信窗口、整理链接和文档、生成 Obsidian 日程卡片。最终发不发送，还是我自己决定。

这才是这套方案最适合主微信的地方。
